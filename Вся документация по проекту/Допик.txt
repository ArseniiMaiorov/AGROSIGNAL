Что фиксируем до старта, чтобы не переделывать


1) Формат и источник ТМИ

Что считаем “истиной”: датасет выдаётся кафедрой/преподавателем (не Kaggle).
Фиксируем как контракт данных (Data Contract):
Формат файла: CSV/Parquet/NPY (что реально выдали) + кодировка + разделитель.
Схема: список признаков (имена, типы), имя целевой колонки (Class или аналог).
Допустимые значения меток: {0,1,2} до возможного объединения.
Пропуски: какие значения считаем пропусками (NaN/пусто/–999/строки), и правило обработки:
удаляем строки / заполняем медианой / forward-fill (если временной ряд) — одно правило, без “в этот раз так, в следующий иначе”.
Частота/время: есть ли явная временная ось (timestamp, номер измерения) и равномерна ли она.
Размер: число строк, число признаков, доля классов.
Отдельно: есть ли неразмеченная часть (нужна для пункта с автоэнкодером).
Артефакт: data_schema.md + src/data/validation.py, которая валит запуск понятной ошибкой, если схема не та.

2) Постановка задачи: бинарная или мультикласс

Связь с методичкой:
Мультикласс (1.1): 0=штатное, 1=отказ, 2=сбой.
Бинарная (1.2): 0=штатное, 1=нештатное (1 и 2 объединяются).
Что фиксируем до кода:
выбранную постановку (1.1 или 1.2);
финальный список меток и их смысл;
правило переразметки (если 1.2): y = 0 if Class==0 else 1.
Артефакт: task_spec.md + проверка в src/data/validation.py, что после переразметки метки ровно {0,1} (для 1.2) или {0,1,2} (для 1.1).

3) Как формируем “последовательность” для GRU
Это главный источник пересборок. Фиксируем один из режимов и не прыгаем.
Режим A — реально временные окна (предпочтительный, если есть time-ось):
последовательность = окно длины T по времени;
вход в модель: (samples, steps=T, channels=features) или (samples, T, features);
окно строим без утечки: разбиение по времени или групповое, если есть “сеансы/пассажи”.
Режим B — псевдопоследовательность признаков (если времени нет):
последовательность = признаки как “шаги”: (samples, steps=features, channels=1);
это учебный приём: работает как архитектурный эксперимент, но честно фиксируем в отчёте, что “время” тут отсутствует.
Что фиксируем численно:
steps (T или число признаков),
способ формирования окон (stride, overlap, padding),
схему входа в Conv1D и GRU (какая ось где).
Артефакт: sequence_spec.md + unit-тесты на shapes: вход → выход конвертера.

4) Обязательность и роль автоэнкодера
По методичке он обязателен, но роль зависит от постановки.
Если постановка 1.1 (мультикласс): автоэнкодер = предобучение на неразмеченных данных (или на данных без меток), потом дообучение классификатора.
Если постановка 1.2 (бинарная): автоэнкодер = аномалия-детект через reconstruction error + порог.
Что фиксируем до реализации:
какой режим (предобучение или аномалия-детект);
откуда берём данные для обучения AE (неразмеченные / весь train без использования меток);
как выбираем порог (например, квантиль ошибки на “норме” в train, или оптимизация по val).
Артефакт: ae_spec.md + тест на синтетике, что аномалии дают большую ошибку восстановления.

5) Целевые пороги по метрикам и ограничения по ресурсам
Чтобы GA и обучение не превратились в бесконечную “варку”.
Метрики (как минимум):
macro-F1 — основной ориентир (особенно при дисбалансе),
balanced accuracy — если дисбаланс заметный,
ROC AUC — считать и обсуждать (для бинарной просто, для мультикласса фиксируем способ: OvR macro).
Цели (реалистично фиксировать так):
“модель по варианту должна обгонять baseline по macro-F1 на валидации минимум на X” (например, +0.03…+0.05),
стабильность: повтор на 3 разных seed не должен “сыпаться” ниже baseline.
Ограничения ресурсов:
железо: CPU/GPU модель (фиксируется в отчёте),
бюджет GA:
population_size, generations,
max_epochs_fitness (малый, например 8–15),
max_epochs_final (больше, например 30–80),
лимит по времени на один запуск (например N часов) — чтобы можно было воспроизвести.
Артефакт: compute_budget.md + параметры в config.yaml.
Итоговый “пакет фиксаций” (чтобы не было пересборок)
До первого серьёзного обучения должны существовать 5 файлов-спецификаций:
data_schema.md
task_spec.md
sequence_spec.md
ae_spec.md
compute_budget.md


И главное: эти решения становятся контрактом проекта. Менять их можно, но только через запись в devlog: “почему меняем” и “какие результаты это улучшило/ухудшило”.


ДОПОЛНИТЕЛЬНЫЕ ПОЯСНЕНИЯ

1) Постановка задачи

Берём 1.1 (multiclass)
Метки: 0 = штатное, 1 = отказ, 2 = сбой.

Почему так: это более строгая постановка (сложнее бинарной), и её проще позже “свернуть” в 1.2, чем наоборот (архитектура/логи/отчёт будут богаче).

2) Режим последовательности для GRU

Берём A: time windows (временные окна)

Фиксируем параметры окна:

T = 128 шагов (длина окна)

stride = 32 шага

overlap = 75% (так как overlap = 1 - stride/T = 1 - 32/128 = 0.75)

Правило формирования:

скользящее окно по времени внутри одного непрерывного ряда/сеанса;

X_window.shape = (samples, 128, n_features)

для 1D-CNN используем Conv1D по оси steps=128, каналы = n_features.

3) Формат и путь датасета

Фиксируем контракт “по умолчанию” (потом заменишь путь на фактический, но логика не меняется):

формат: CSV

путь: data/tmi.csv

разделитель: ,

кодировка: utf-8

null-токены (считаем пропусками):

пустая строка ""

NA, N/A, NaN, nan, null, None

-999, -9999 (типовые заглушки датчиков)

Важно: это именно “контракт загрузчика”. Если у тебя окажется ; или другая кодировка — меняется только конфиг, не код.

4) Compute budget и целевой прирост качества

Фиксируем бюджет так, чтобы GA был реальным, а не “вечным”:

population_size = 12

generations = 8
→ всего 96 оценок (12×8), это нормальный компромисс.

max_epochs_fitness = 10 (короткий прогон для fitness)

max_epochs_final = 50 (финальный прогон лучшей конфигурации)

Обязательные стопы (чтобы не жечь ресурсы):

EarlyStopping(patience=5, restore_best_weights=True)

ReduceLROnPlateau(patience=3, factor=0.5)

Цель по качеству:

X = +0.03 к macro-F1 относительно baseline на validation (не на test).

Готовый блок конфига (вставляй как config.yaml)
task:
  type: multiclass   # 1.1
  labels: [0, 1, 2]
  target_col: "Class"

sequence:
  mode: time_windows # A
  T: 128
  stride: 32
  overlap: 0.75

data:
  format: csv
  path: data/tmi.csv
  sep: ","
  encoding: "utf-8"
  null_tokens: ["", "NA", "N/A", "NaN", "nan", "null", "None", -999, -9999]

compute_budget:
  population_size: 12
  generations: 8
  max_epochs_fitness: 10
  max_epochs_final: 50
  target_macro_f1_gain_vs_baseline: 0.03

training:
  early_stopping_patience: 5
  reduce_lr_patience: 3
  reduce_lr_factor: 0.5

Это фиксирует все “развилки”, которые обычно вызывают пересборки. Дальше можно спокойно пилить код: загрузка → окна → baseline → hybrid → GA → финал → отчёт.
